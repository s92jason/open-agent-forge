# Chapter Rules

## 核心原則：Single Source of Truth per Dimension

每條規則只在一個章節完整定義，其他章節只引用，不重複展開：

| 資訊維度             | 唯一完整定義位置          | 其他章節的引用方式   |
|------------------|-------------------|-------------|
| 欄位定義（型別、用途）      | 🔌 API 規格         | 寫欄位名稱，不展開定義 |
| 欄位對 UI 的影響       | 📐 業務規則 或 📱 畫面規格 | 「見📐業務規則」   |
| 欄位在流程中的角色        | 🔄 核心流程           | Mermaid 圖表達 |
| 完整型別 pseudo-code | 附錄 A              | 「完整型別見附錄 A」 |

**違反判定**：若同一欄位的行為邏輯在兩個以上章節各自展開超過一行，即違反此原則。

---

## 文件區域設計

FEATURE-SPEC.md 分為兩個區域：

**Human Zone**：讓人類工程師快速建立功能認知

- 📋 摘要
- 🎯 目標與範圍
- 🔄 核心流程
- 📐 業務規則
- 📱 畫面規格
- 🔌 API 規格
- 🛠️ 程式影響範圍
- ✅ 驗收條件
- ❓ 待確認事項（條件性）

**AI Reference Zone**：讓 AI Agent 查找結構化工程資料

- 附錄 A：型別定義與欄位對應
- 附錄 B：歷史與遷移參考
- 附錄 C：參考資料與變更紀錄

## Document Status

必備。以文件開頭 header metadata 表達，不使用獨立標題。

### 必填

- `# Feature Spec: <Feature Name>`
- `Status: <Draft | Pending API confirmation | In Progress | Ready>`
- `Last Updated: <YYYY-MM-DD>`
- `Mode: <Create | Update>`
- `Structure: Human Zone (摘要 → 驗收條件) | AI Zone (附錄 A～C)`

### 狀態建議

- Draft：來源不足但先出草稿
- Pending API confirmation：API 未定
- In Progress：已有主要來源但仍有少量待補
- Ready：已確認且可交付實作

## 📋 摘要

讓 RD 在前 30 秒建立功能認知。**必須先說功能「是什麼」，再說「這次改了什麼」**。

### 結構（依序）

1. **功能簡介**（必備）：2-3 行說明這個功能的目的與定位，讓第一次接手的人能理解。不是說「這次改了什麼」，而是說「這東西幹嘛用的」。
2. **這次改了什麼**：一段話，精簡描述本次改版重點。
3. **使用者主路徑**：一行，描述 happy path。
4. **前端負責 vs 不負責**：表格，清楚劃出前端邊界。
5. **影響的 API / 模組 / 畫面**：條列清單。

### 不要寫

- PM 故事、空泛描述
- 完整的 scope 說明（那是🎯目標與範圍的工作）

### Pending Summary

若有待補，放在📋摘要之後、🎯目標與範圍之前：

```
- [Pending] API aggregation 欄位命名待確認
- [Conflict] Axure retry flow 與 Figma dialog action 不一致
```

| 頁面             | 說明    | 進入方式 |
|----------------|-------|------|
| `<ScreenName>` | 一句話說明 | 如何進入 |

### 主流程圖

### 替代流程與失敗處理

## 📐 業務規則

整份文件的跨章節引用核心。其他章節提到同一規則時，引用此章而不重寫。

### 寫法

依規則類型分子節，每子節用表格呈現。**每張規則表必須包含「實作位置」欄位**。

```
| 情境 | 條件 | 行為 | 實作位置 |
|------|------|------|----------|
| xxx  | xxx  | xxx  | TBD      |
```

- `實作位置` 初始為 `TBD`
- 功能開發完成後，掃過 code 回填檔案名稱與路徑（如 `FeatureOverviewBlock.kt:L42`）
- 這讓其他人能直接從規則表跳到對應程式碼

### 補充說明

- 若有已廢棄的舊邏輯，不在此章說明，移到附錄 B
- 術語定義（Glossary）放在附錄 A，不放在此章

## 📱 畫面規格

每個畫面一個完整區塊。

### 每個畫面的結構

```
### <ScreenName>（中文名稱）

**用途**：一句話

**資料來源**：API 名稱 + 頂層物件名稱（不展開完整欄位，完整定義在🔌 API 規格）
- 需特殊處理的欄位：只列出「不看 spec 會搞錯」的欄位

**狀態**
| 狀態 | 觸發 | 畫面行為 |
|------|------|----------|

**使用者操作**
- 操作 → 結果（涉及業務規則時引用📐業務規則）

**元件細節**（需要時）
- 具體元件描述

**錯誤處理**（需要時）
- fallback 行為
```

### 必看點

- Loading / Success / Empty / Error
- enabled / disabled / hidden
- retry / refresh / back

### 不要寫

- 像素級視覺描述
- 在此章重新定義欄位的 UI 邏輯（那是📐業務規則的工作）

## 🔌 API 規格

**欄位定義的唯一完整來源**。其他章節引用欄位時只寫名稱，不展開定義。

### 每支 API 的結構

```
### <API_NAME>（中文名稱）

**用途**：一句話
**呼叫時機**：何時呼叫

**Request**
| 欄位 | 型別 | 說明 |

**前端使用的回傳欄位**
| 欄位 | 型別（選填） | 用途 |

**前端不使用的欄位**
| 欄位 | 原因 |

**備註**：（選填）
```

### 關鍵規則

- 明確區分「前端使用」與「前端不使用」兩張表，每個不使用的欄位都要寫原因
- 完整型別定義（pseudo-code）放在附錄 A.2，主文不列
- API 未定時保留章節，標示 `[Pending API confirmation]`

### 不要寫

- Swagger 全貼
- 在此章描述欄位對 UI 的影響（那是📐業務規則或📱畫面規格的工作）
